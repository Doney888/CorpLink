# CorpLink — Система управления корпоративными коммуникациями

**CorpLink** — это высоконадежная бэкенд-платформа для мгновенного обмена сообщениями внутри закрытого контура организации. Проект реализует полноценную микросервисную архитектуру, ориентированную на безопасность данных, отказоустойчивость и строгий аудит действий пользователей.

Проект является совмещенной выпускной работой по дисциплинам **«Проектирование и реализация баз данных»** и **«Объектно-ориентированное программирование»** (Университет ИТМО, ПИН/ИКТ).

## Архитектура системы

Бэкенд декомпозирован на два изолированных микросервиса, взаимодействующих по сетевым протоколам (паттерн *Database-per-Service*):

```text
                               ┌───────────────────┐
                               │   API Gateway     │ (Port 8080)
                               └─────────┬─────────┘
                                         │ (Routing)
                    ┌────────────────────┴────────────────────┐
                    ▼                                         ▼
        ┌───────────────────────┐                 ┌───────────────────────┐
        │   Identity Service    │ (Port 5001)     │     Chat Service      │ (Port 5002)
        └───────────┬───────────┘                 └───────────┬───────────┘
                    │                                         │
        ┌───────────┴───────────┐                 ┌───────────┴───────────┐
        │  PostgreSQL (users)   │                 │ PostgreSQL (messages) │
        │  Redis (sessions)     │                 │                       │
        └───────────────────────┘                 └───────────────────────┘
```

1.  **Identity Service:** Регистрация, авторизация (JWT), профили сотрудников.
2.  **Chat Service:** Управление комнатами, участниками, отправка и редактирование сообщений, логирование правок.

## Физическая модель данных (PostgreSQL)

Реляционная часть СУБД приведена к **3НФ** и состоит из **10 таблиц** (DDL-дамп расположен по адресу `database/corplink_init_db.sql`).

*   **Суррогатные ключи:** Все первичные ключи (`PK`) объявлены как `UUID` (`gen_random_uuid()`) для безболезненного горизонтального масштабирования (шардирования) баз данных в будущем.
*   **Целостность и каскады:** Связующая таблица участников `chat_members` имеет ограничение `UNIQUE(chat_id, user_id)` для защиты от дублирования. Настроены каскадные правила `ON DELETE CASCADE` для автоматической очистки зависимого контента.
*   **Аналитические Представления (Views) (ЛР 4):**
    *   `v_chat_members_details` — денормализованные данные об участниках чатов и их ролях.
    *   `v_unread_messages_count` — расчет количества непрочитанных сообщений по чатам индивидуально для каждого пользователя.
*   **Индексация (ЛР 4):** Составной B-Tree индекс `idx_messages_chat_sent_at` на `(chat_id, sent_at DESC)` оптимизирует загрузку истории чата, полностью исключая ресурсоемкую операцию сортировки `Sort` в ОЗУ.
*   **Хранимый программный слой (ЛР 5):**
    *   Процедура `sp_create_group_chat` — транзакционное создание чата и автоматическое добавление создателя в качестве Владельца (Owner).
    *   Процедура `sp_get_chat_statistics` — использует параметры `IN` и `OUT` для быстрого сбора аналитики чата.
    *   Триггер `trg_audit_message_edits` (`BEFORE UPDATE OR DELETE`) — автоматически фиксирует историю изменений текста в таблицу `message_audit_logs`, используя системную переменную `TG_OP`.
    *   Триггер `trg_check_sender_in_chat` — валидирует "узкое место" безопасности, запрещая отправку сообщений не-участникам чата.

## Интеграция NoSQL (Redis)

Для работы с высокочастотными временными данными используется In-Memory NoSQL СУБД Redis:
*   `user:presence:{id}` (TTL = 300) ➔ `"online"` (статус сотрудника в сети).
*   `chat:{id}:typing:{id}` (TTL = 5) ➔ `1` (индикация ввода текста).
*   `token:blocklist:{jti}` (TTL = 3600) ➔ `"revoked"` (черный список JWT-токенов).

## Спецификация REST API и Межсервисного взаимодействия

### Identity Service (Port 5001)
*   `POST /api/auth/register` — Регистрация нового сотрудника (`201 Created`).
*   `POST /api/auth/login` — Вход в систему, выдача токена (`200 OK`).
*   `GET /api/users/{id}/verify` — Синхронная проверка существования пользователя для Chat Service.

### Chat Service (Port 5002)
*   `POST /api/chats` — Создать групповой чат (вызывает процедуру СУБД).
*   `GET /api/chats/{chatId}/messages` — Получить историю сообщений (выборка из индекса).
*   `POST /api/chats/{chatId}/messages` — Отправить сообщение (выполняет синхронный запрос в Identity Service и вызывает Mock-сервис уведомлений).

## Инструкция по локальному запуску

### 1. Подготовка Баз данных
Импортируйте SQL-дамп из папки `database/corplink_init_db.sql` в вашу базу данных PostgreSQL `corplink_db` на порту `5432`. Убедитесь, что Redis-сервер запущен на стандартном порту `6379`.

### 2. Запуск микросервисов
Из корня проекта запустите независимые деплойменты через Gradle:

```bash
# Запуск сервиса авторизации (Port 5001)
./gradlew :identity-service:run
# Запуск сервиса чатов (Port 5002)
./gradlew :chat-service:run
```

### 3. Запуск интерактивной панели мониторинга (Python)
Установите зависимости и запустите Streamlit для визуального тестирования СУБД:
```bash
pip install -r requirements.txt
streamlit run app.py
```

*Админка откроется по адресу http://localhost:8501. Схемы всех диаграмм (UML, Use Case, ERD) расположены в папке `database/diagrams/`.*
